Organizing Data(重新组织数据)

2016-07-02

这是《Refactoring》的第8章，总共讲了有18种方法来重构你的代码，均是针对类的field来谈的

1.Self Encapsulate Field（自封装字段）

直接访问字段时会使代码和字段耦合得很紧密，可以通过getter和setter的方法来访问字段

2.Replace Data Value with Object（以对象取代数据值）

当项目由简单变复杂后，由于访问的字段多，可将其单独拧出一个类来。

3.Change Value To Reference（将值对象改为引用对象）

改变对象后所有对这个对象的使用都需要改变时应该使用引用对象。

4.Change Reference To Value（将引用对象改为值对象）

书中说该类型在并发编程中用得比较多。

5.Replace Array with Object（以对象取代数组）

书中认为用数组访问需要记住每个字段对顺序，其实不是有用代表字段意义的字符串做键了吗？感觉书中所说的理由不够充分，我认为应该是当取数据时，如果没有此下标的话数组会报错，而对象在很多编程语言里一般不会（法克，我用的语言也不多），选择适合的场景吧

6.Duplicate Observed Data（复制被监视的数据）

分成好的系统是将用户页面和处理业务的逻辑分开的，但是两边的数据却不是那么容易分开，数据需要保存在两边，这时可以创建observer模式来同步两边的数据。

7.Change Unidirectional Association To Bidirectional（将单向关联改为双向关联）

8.Change Bidirectional Association To Unidirectional（将双向关联改为单向关联）

10.Encapsulate Field（封装字段）

11.Encapsulate Collection（封装集合）

集合应该以不可修改的方式返回？why，有待研究

12.Replace Record with Data Class（以数据类取代记录）

创建一个数据类供类用，使程序更面相对象

13.Replace Type Code with Class（以类取代类型码）

类型码降低了代码的可读性，方面扩展。这理由不能说服我呀。还是直接copy代码吧

14.Replace Type Code with Subclasses（以子类取代类型码）

不可改变的类型码会影响到类的行为，可以将类型码替换成子类，也可以用多态，但在两种情况下不可以使用该方法。1:类型码是在类创建后改变的，2:类型码已经在子类里了

15.Replace Type Code with State/Strategy（以state/strategy取代类型码）

使用场景：在对象的周期中变化时或不能使用子类时。

16.Repalce Subclass with Fields（以字段取代子类）

当子类只有静态方法时，应该用字段去取代它。避免过度封装
