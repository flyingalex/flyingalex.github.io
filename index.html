Making Method Calls Simpler（简化函数调用）

2016-07-11

1.Rename Method（函数改名）

名字应该表达函数体的意思，取名时可根据注释来取

2.Add Parameter（添加参数）

书中不建议这么做，参数太长影响代码可读性，可查看是否可以通过现有的参数传递，或者其他可替代的方法。

3.Remove Parameter （移出参数）

删除不再被方法使用的参数

4.Separate Query From Modifier （将查询函数和修改函数分离）

一个方法既返回值也改变对象的状态时应该将其拆分成两个不同的方法。单一指责的原则。

5.Parameterize Method （令函数携带参数）

几个方法非常相似，只是因为包含几个不同的值，可将其抽成一个方法，不同的值通过参数的方法传递进去。

6.Replace Parameter with Explicit Methods（以明确函数取代参数）

当一个方法根据参数不同的值执行不同的代码段时可以为每一个不同的值创建一个方法

7.Preserve Whole Object（保持对象完整）

当传递给方法的不同参数来至同一个对象，可对象直接传入方法，保证对象的完整性，亦方法后面扩展。当然如果一个方法使用对象上的很多属性或方法，这时得考虑这个对象的存放位置了。

8.Replace Parameter with Method（以对象取代参数）

当一个对象调用方法获得的结果作为另外一个方法的参数时，可直接将参数换成函数的调用。

9.Indroduce Parameter Object（引入参数对象）

当有一组参数时，可引入对象来包含参数，不仅能增加代码的可读性，而且可将一些修改参数的方法也一并方法对象，减少重复的代码。

10.Remove Setting Method（移除设值函数）

当参数值只有在构造函数设置时，其他时候不改，可将设置方法去掉。

11.Hide Method（隐藏函数）

方法不被其他类使用时，应该将其隐藏掉。

12.Replace Constructor with Factory Method（以工厂函数取代构造函数）

当需要创建复杂的类时，可用工厂方法替代构造函数。

13.Encapsulate Downcast（封装向下转型）

当方法放回的对象需要转型时，尽量在方法内转型

14.Replace Error Code with Exception（以异常取代错误码）

用异常取代错误码

15.Replace Exception with Test（以测试取代异常）

用测试取代异常，有的条件是需要先检验的。

Dealing with Generalization（处理概括关系）

2016-07-10

1.Pull Up Field（字段上移）

将子类中相同的字段移入父类

2.Pull Up Method（函数上移）

子类中的方法获得的结果一样，可以提取到父类

3.Pull Up Constructor Body（构造函数本体上移）

当子类中的构造函数与父类有很多相同的地方，可以将子类与父类相同的部分移除，调用父类的方法

4.Pull Down Method（方法下移）

方法应该只放在属于它的类里

5.Pull Down Field（字段下移）

同上，字段应该只放在属于它的类里

6.Extract Subclass（提炼子类）

当一个类中的属性只是属于一些实例时，应该创建子类来分离这些属性

7.Extract Superclass（提炼超类）

当两个类很相似时可以提取超类，减少重复对代码

8.Extract Interface（提炼接口）

9.Collapse Hierarchy（折叠继承体系）

根据实际条件修改

10.Form TemPlate Method（塑造模版方法）

当子类中有两个方法做相似的事情，只是步骤有些不同，将这些步骤统一命名，燃火提取方法

11.Replace Inheritance with Delegation（以委托取代继承）

当子类只需要父类的少许功能时，可以通过代理来实现

12.Repacle Delegation with Inheritance（以继承取代委托）

当子类中代理太多时，可考虑用继承取代代理

Moving Features Between Objects(在对象间搬移特性)

2016-07-01

1.Move Method（搬移函数）

当一个方法更多的使用的是其他类的信息，而不是定义的类里的信息，可以将其抽到使用更多信息的类里。保证类单一职责的原则进行操作

2.Move Field（搬移字段）

当定义在类A中的字段更多的被B类使用时，应该将其移入B类中.

3.Extract Class（提炼类）

当一个类职责不明时，观察类的哪些属性和方法可放在一起形成一个单一职责的类，按此拆分。

4.Inline Class（类内联化）

与上面的“提炼类”是相反的操作，书中的说法我不是很理解，英语理解能力（卧槽。。。），我的理解时，当类A只被类B使用时，可以将类A移入类B中

5.Hide Delegate（隐藏委托关系）

当客户端代码需要通过一个代理去调用一个方法时，服务端可创建一个方法来隐藏代理。这样做是为了更好的封装，客户端只需要知道与之相关的类，减少对其他类的依赖。

6.Remove Middle Man（移除中间人）

与上面“隐藏委托关系”的操作相反。这两种操作可根据系统的变化而变化。

7.Introduce Foregin Method（引入外加函数）

当使用服务端的类需要另外添加一个方法但不能修改服务端类时可引入外加函数。特别当有几个地方都需要用时，添加一个方法起到封装的作用。

8.Introduce Local Extension（引入本地扩展）

当使用服务端的类需要另外添加几个方法但不能修改服务端类时可引入外本地扩展。可选择封装或继承。

Composing Methods(重新组织函数)

2016-06-29

这是Refactoring的第六章，讲了如何重新组织让代码更加简洁，目标感觉就是用小方法代替大方法,下面对书中的方法做一终结。

1.Extract Method (提炼方法)

动机：方法太长或代码需要注释去理解时可将那部分变成一个方法

2.Inline Method (内联函数)

当函数体能清楚的表达名字的意思的时候，一般是小方法（函数体代码量小），这时的代理一般是不必要的，可以用内联函数代替

3.Inline Temp (内联临时变量)

方法内的临时变量只是用于存储一个简单表达式的值，可以将其直接用表达式替换

4.Replace Temp with Query (以查询取代临时变量)

临时变量只在只存在于方法内，不可在方法外复用。如果将这些临时变量用方法来替换，类里的任何方法都可以获得它

5.Introduce Explaining Variable (引入解释性变量)

用一个名字来代替复杂表达式所要表达的意思，一般用在条件判断或算法中

6.Split Temporary Variable (分解临时变量)

用于一个变量被赋值两次的情况，不包含循环语句或集合用临时变量，因为一个变量表示多个意思很容易误导他人。

7.Remove Assignments to Parameters (移除对参数的赋值)

用临时变量替换，在传参时有时是传值有时是传引用，因而直接修改参数时可能出现问题

8.Repalce Method with Method Object (以函数对象取代函数)

可将本地变量都变成函数对象的字段，将一个大方法分解成多个小方法

9.Substitute Algorithm (替换算法)

用简单算法替换复杂掉算法（卧槽，这个如果知道的话肯定会替换的啊）

代码的坏味道

2016-06-26

一直觉得基础是非常重要的，自己一直也在补充这方面的知识，最近在读《Refactoring–Improving the Design of Existing Code》，读到第6章了，觉得有必要将其中的一些知识点写一下，以便加深印象。开始吧。。。。

前面几章我觉得第三章比较重要——代码的坏味道（bad smells in code），就从这章开始吧

问题: 什么时候开始重构你的代码？

重构22条:

1.重复的代码(dumplicated code)

相同或相似的代码，将相同的地方抽取出来，避免重复的代码。如果代码是重复的，当修改一个地方，就需要修改其他地方，做无用功，有时如果忘了修改其中一个地方就悲剧了，因此相同的代码尽量抽出来放一个地方便于维护。

2.长方法（long method）

方法太长不宜理解。有太多的参数和临时变量可以抽出来。判断和循环也可抽出来。用小方法代替长方法

3.大类(large class)

当一个类很大时，一般这样的类里都有很多实例变量。可以将一些实例变量抽成另外一个类。

4.长参数列表（long parameter list）

可用传对象的方式改变参数的长度，并且具有更好的维护性和可读性

5.发散式变化（divergent change）

当一个类因为不同的原因以不同的方式变化。应该将其切分成几个小点方法，每个类只因为一种变化而改变。

6.散弹式变化（shotgun surgery）

当发生一种变化时，同时有几个类度都需要做修改。这时因将全部的修改集中到一个类中

7.依恋情节（feature envy）

类里的方法更多的和另外的类交互，最多的例子是一个类里有很多的get方法与其他类相关。应该根据数据和哪个类最相关就应将其放在哪个类。

8.数据泥团（data clumps）

很多数据经常出现在一起，可以将它们打包成一个对象

9.基本类型偏执（primitive obsession）

编程时总喜欢用基本类型，而不喜欢用对象。增加扩展和修改的复杂性

10.switch 状态

使用switch不好的地方在于增加一个条件的时候需要修改switch语句。在oop中可以多态的方式来解决

11.平行继承体系（parallel inheritance hierarchies）

平行继承体系是散弹式变化（shotgun surgery）的一种特殊例子，当为一个类增加一个子类时，就要为另外一个类添加一个子类。可通过让一个类引用另外一个类的实例来避免这种情况

12.懒类（lazy class）

一个类如果能做的事不如创建它所耗费的多就应该消除掉它，或减少它的数量，一些无用的类也可以用行内类代替

13.夸夸其谈未来性（Speculative Generality）

定义将来可能需要的方法或其他，应该将此删除

14.临时字段（Temporary Field）

一个类里的有些变量只在某些环境中有作用，这时应该将它们提取出来做为一个新的类

15.消息链（message chains）

当客户端调用一个类去访问另外一个类时，另外一个类又去访问其他的类，如此反复，当中间的任何类变化都有可能影响到客户端的代码。一般的做法是隐藏中间类。更好的做法是看最终使用的对象是什么，看看是否可以将其抽出来放在一个方法中供客户端代码调用。

16.中间人（middle man）

当一个类中的接口大部分都是其他类的代理时可以考虑将其抽出来，让其与那个紧密联系的类交互。另外可以行内方法或继承重构。

17.不合时宜的紧密关系（inappropriate intimacy）

有时候有些类太过紧密，大部分时间都在处理对方的私有部分，这时应该通过迁移方法和子段来将其分开，减少这种紧密联系。继承很容易造成这种情况，可以用代理来代替继承

18.有不同接口的可选类（alternative classes with different interfaces）

将做同样事情的类统一命名，并移动代码直到一个类遵守的协议相同为止。

19.不完善的库类（incomplete library class）

修改一个库类来达到我们想要的要求一般不可能。当然可以通过引入外加函数和引入本地扩展来达到目的。

20.数据类（data class）

数据类知道很多数据的细节，并且被其他类使用，应该将数据封装起来，将setter和getter方法移入数据类，删除不必要的setter和getter方法

21.被拒绝的遗赠（refused bequest）

子类复用超类的行为但是拒绝支持超类的接口，这时可用代理来替换继承

22.评论（comments）

当需要注释去解释一段代码需要做什么的时候，尝试将其抽成一个方法。当抽出方法后仍需要注释的话可以尝试修改方法名。
